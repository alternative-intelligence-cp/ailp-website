Ultra-Large Signed Integers (int1024, int2048, int4096)

Signed multi-precision integers for cryptographic operations and specialized mathematical research

Overview

Aria provides three ultra-large signed integer types that extend far beyond standard CPU register sizes:

| Type | Bits | Range (approximate) | Bytes | Decimal Digits | Primary Use Cases | |------|------|---------------------|-------|----------------|-------------------| | int1024 | 1024 | ±8.9×10³⁰⁷ | 128 | ~308 | RSA-1024 (deprecated), multi-precision math | | int2048 | 2048 | ±3.2×10⁶¹⁶ | 256 | ~617 | RSA-2048 crypto, secure key generation | | int4096 | 4096 | ±1.0×10¹²³³ | 512 | ~1,233 | RSA-4096 (high security), post-quantum prep |

CRITICAL CONTEXT: These types exist for cryptographic operations and specialized mathematical research only. They are:
  • Software-implemented (no CPU hardware support)
  • Extremely slow compared to native types
  • Memory-intensive (128-512 bytes per value)
  • Specialized tools - not general-purpose integers!

Why Ultra-Large Signed Integers?

Cryptographic Requirements

Modern cryptography requires arithmetic on very large numbers: 

// RSA key generation requires large prime arithmetic


// Private key operations use signed modular arithmetic
fn generate_rsa_key(bits: int) -> (int4096, int4096) {
let p = generate_large_prime(bits / 2);  // Large prime
let q = generate_large_prime(bits / 2);  // Another large prime
let n = p * q;                            // Public modulus
let phi = (p - 1) * (q - 1);             // Euler's totient (signed!)




// Compute private exponent using extended Euclidean algorithm
let d = mod_inverse(65537, phi);         // Needs signed arithmetic




return (n, d);
}

Why signed?
  • Extended Euclidean algorithm produces negative intermediate values
  • Modular inverse calculations require signed arithmetic
  • Some crypto protocols use signed representations

Number Theory Research

// Riemann hypothesis computations


fn compute_zeta_zero(precision: int4096) -> flt512 {
// Number theory often needs ultra-high precision integers
// for intermediate calculations
let numerator: int4096 = ...;
let denominator: int4096 = ...;
return flt512(numerator) / flt512(denominator);
}

int1024: 1024-bit Signed Integer

Specifications

  • Size: 128 bytes (1024 bits)
  • Range: -2¹⁰²³ to 2¹⁰²³ - 1 (±8.98×10³⁰⁷)
  • Precision: ~308 decimal digits
  • Alignment: 16-byte (128-bit) aligned for cache efficiency
  • Two's complement representation

Memory Layout

Byte:  0      8      16     24    ...   120   127


[------|------|------|------...-----|------]
LSB                              MSB (sign)

128 bytes = 2 cache lines on most CPUs (severe cache pressure!)

When to Use int1024

// ✓ GOOD: Legacy RSA-1024 (though now deprecated for security)


fn decrypt_rsa1024(ciphertext: uint1024, d: int1024, n: uint1024) -> uint1024 {
// Private key exponent can be signed in some implementations
return mod_exp(ciphertext, d, n);
}




// ✓ GOOD: Multi-precision arithmetic intermediate values
fn factorial_1024bit(n: uint64) -> int1024 {
let result: int1024 = 1;
till(n, 1) {
result *= int1024($);
if result == ERR { return ERR; }  // Overflow check
}
return result;
}




// ✓ GOOD: Cryptographic protocol signatures (some schemes)
fn sign_message(msg_hash: int1024, private_key: int1024) -> int1024 {
// Some signature schemes use signed arithmetic
return mod_exp(msg_hash, private_key, MODULUS);
}




// ✗ BAD: Using as general counter (absurd overkill!)
let counter: int1024 = 0;  // NO! Use int64!
counter += 1;




// ✗ BAD: Arrays of int1024 (memory explosion!)
let array: [int1024; 1000];  // 128 KB! Use smaller types!




// ✗ BAD: Hot loop operations (glacially slow!)
till(999_999, 1) {
let x: int1024 = compute();  // Each op is ~100× slower than int64!
}

Performance Characteristics

// Rough performance comparison (relative to int64 = 1×)


let x: int64 = 12345;
let y: int64 = 67890;
let z = x * y;  // ~1 cycle (native CPU instruction)




let a: int1024 = 12345;
let b: int1024 = 67890;
let c = a * b;  // ~100-500 cycles! (software multi-precision multiply)

Reality check: int1024 operations are ~100-500× slower than int64:
  • Addition: ~100× slower (multi-word carry propagation)
  • Multiplication: ~200-500× slower (Karatsuba or similar algorithm)
  • Division: ~500-2000× slower (multi-precision long division)
  • Modular exponentiation: Seconds per operation!

Cryptographic Example

// Extended Euclidean Algorithm (needs signed arithmetic!)


fn extended_gcd(a: int1024, b: int1024) -> (int1024, int1024, int1024) {
if b == 0 {
return (a, 1, 0);  // gcd, x, y where ax + by = gcd
}




let (gcd, x1, y1) = extended_gcd(b, a % b);




// These intermediate values can be NEGATIVE!
let x = y1;
let y = x1 - (a / b) * y1;




return (gcd, x, y);
}




// Modular multiplicative inverse (private key computation)
fn mod_inverse(a: int1024, m: int1024) -> int1024 {
let (gcd, x, _) = extended_gcd(a, m);




if gcd != 1 {
return ERR;  // No inverse exists
}




// x might be negative, normalize to positive
return (x % m + m) % m;
}

int2048: 2048-bit Signed Integer

Specifications

  • Size: 256 bytes (2048 bits)
  • Range: -2²⁰⁴⁷ to 2²⁰⁴⁷ - 1 (±3.23×10⁶¹⁶)
  • Precision: ~617 decimal digits
  • Alignment: 32-byte (256-bit) aligned
  • Current standard for RSA cryptography

Memory Layout

256 bytes = 4 cache lines (MASSIVE cache footprint!)


Array of 1,000: 256 KB (entire L2 cache on some CPUs!)
Array of 1M: 256 MB (won't fit in cache at all!)

When to Use int2048

// ✓ GOOD: RSA-2048 cryptography (current industry standard)


fn rsa2048_sign(message: int2048, d: int2048, n: int2048) -> int2048 {
// Signature = message^d mod n
// This operation can take MILLISECONDS!
return mod_exp(message, d, n);
}




// ✓ GOOD: Elliptic curve operations (some curves)
fn ec_point_multiply(scalar: int2048, point: ECPoint) -> ECPoint {
// Some elliptic curves operate on 2048-bit fields
// ...
}




// ✓ GOOD: Large prime generation
fn generate_2048bit_prime() -> int2048 {
loop {
let candidate = generate_random_int2048();
if is_probably_prime(candidate, 40) {  // Miller-Rabin
return candidate;
}
}
}




// ✗ BAD: Storing as array elements (memory disaster!)
struct KeyPair {
public_key: int2048,   // 256 bytes
private_key: int2048,  // 256 bytes
}
let keys: [KeyPair; 10000];  // 5 MB for only 10K keys!




// ✗ BAD: Frequent arithmetic operations
till(999, 1) {
let x: int2048 = expensive_compute();  // Each op takes microseconds!
}

RSA-2048 Complete Example

// Complete RSA-2048 key generation and signing


struct RSA2048KeyPair {
n: uint2048,      // Public modulus
e: uint64,        // Public exponent (typically 65537)
d: int2048,       // Private exponent (SIGNED!)
p: uint2048,      // Secret prime 1
q: uint2048,      // Secret prime 2
}




fn generate_rsa2048_keypair() -> RSA2048KeyPair {
// Generate two random 1024-bit primes
let p = generate_1024bit_prime();
let q = generate_1024bit_prime();




// Public modulus n = p * q
let n = uint2048(p) * uint2048(q);




// Euler's totient: φ(n) = (p-1)(q-1)
// This requires SIGNED arithmetic!
let phi = int2048(p - 1) * int2048(q - 1);




// Public exponent (standard)
let e: uint64 = 65537;




// Private exponent: d = e^(-1) mod φ(n)
// Extended Euclidean algorithm produces SIGNED values!
let d = mod_inverse(int2048(e), phi);




if d == ERR {
panic("Failed to compute modular inverse!");
}




return RSA2048KeyPair { n, e, d, p, q };
}




fn rsa2048_encrypt(plaintext: uint2048, e: uint64, n: uint2048) -> uint2048 {
// Encryption: c = m^e mod n
// This is FAST (e is small, only 65537)
return mod_exp_fast(plaintext, uint2048(e), n);
}




fn rsa2048_decrypt(ciphertext: uint2048, d: int2048, n: uint2048) -> uint2048 {
// Decryption: m = c^d mod n
// This is SLOW (d is huge, ~2048 bits)
// Can take MILLISECONDS per operation!
return mod_exp(ciphertext, d, n);
}

Performance Reality Check

// Benchmark: RSA-2048 operations


fn benchmark_rsa2048() {
let keypair = generate_rsa2048_keypair();  // ~500ms to 2 seconds!




let message: uint2048 = 42;




let start = get_timestamp();
let ciphertext = rsa2048_encrypt(message, keypair.e, keypair.n);
let encrypt_time = get_timestamp() - start;  // ~0.1-1 ms




let start = get_timestamp();
let decrypted = rsa2048_decrypt(ciphertext, keypair.d, keypair.n);
let decrypt_time = get_timestamp() - start;  // ~10-50 ms! (much slower!)




println("Encryption: &{encrypt_time} ms");
println("Decryption: &{decrypt_time} ms");  // 10-100× slower than encryption!
}

Why decryption is slow: Private exponent d is ~2048 bits, requiring ~2048 modular multiplications (square-and-multiply algorithm).

int4096: 4096-bit Signed Integer

Specifications

  • Size: 512 bytes (4096 bits)
  • Range: -2⁴⁰⁹⁵ to 2⁴⁰⁹⁵ - 1 (±1.04×10¹²³³)
  • Precision: ~1,233 decimal digits
  • Alignment: 64-byte (512-bit) aligned
  • High-security cryptographic standard

Memory Layout

512 bytes = 8 cache lines (ABSURDLY MASSIVE!)


Single value: Half a kilobyte!
Array of 1,000: 512 KB (won't fit in L2 cache!)
Array of 1M: 512 MB (absurd memory usage!)

When to Use int4096

// ✓ GOOD: RSA-4096 (high-security, government/military)


fn rsa4096_generate_keypair() -> (uint4096, int4096, uint4096) {
let p = generate_2048bit_prime();
let q = generate_2048bit_prime();
let n = uint4096(p) * uint4096(q);
let phi = int4096(p - 1) * int4096(q - 1);
let d = mod_inverse(int4096(65537), phi);
return (n, d, uint4096(65537));
}




// ✓ GOOD: Post-quantum cryptography preparation
fn pqc_precompute_parameters() -> int4096 {
// Some post-quantum schemes need ultra-large integer arithmetic
// for parameter generation
}




// ✓ GOOD: Extreme-precision number theory
fn compute_massive_factorial(n: uint32) -> int4096 {
// Research computations requiring ~1000+ digit precision
}




// ✗ BAD: Using in hot paths (each op takes MILLISECONDS!)
till(999, 1) {
let x: int4096 = compute();  // DISASTER! Loop takes SECONDS!
}




// ✗ BAD: Allocating arrays
let primes: [int4096; 100];  // 50 KB for only 100 values!




// ✗ BAD: Storing in structs frequently accessed
struct Config {

}

RSA-4096 Performance

// RSA-4096 operations are EXTREMELY slow


fn benchmark_rsa4096() {
// Key generation: Can take 10-60 SECONDS!
let start = get_timestamp();
let keypair = generate_rsa4096_keypair();
let keygen_time = get_timestamp() - start;
println("Key generation: &{keygen_time} seconds");  // 10-60 seconds!




// Encryption: Still relatively fast (~1-10 ms)

// Decryption: VERY slow (100-500 ms per operation!)

Reality: RSA-4096 operations are so slow they're only used when security absolutely requires it:
  • Government/military communications
  • Long-term secret storage (10+ year security horizon)
  • Certificate authorities (CA root keys)
  • Post-quantum transition period

Common Operations

Arithmetic Operations

All operations are software-implemented and extremely slow:

// Multiplication (~200-1000× slower than int64)

// Division and modulo (~500-5000× slower than int64)

let remainder = a % b;  // Same algorithm as division




// Overflow detection (all operations check!)
if product == ERR {
println("Multiplication overflow!");
}

Comparison Operations

let a: int2048 = 99999;


let b: int2048 = 100000;




// Comparisons (~10-50× slower than int64)

Bitwise Operations

let a: int2048 = 0xFF00FF00;


let b: int2048 = 0x00FF00FF;

// Shifts involving word boundaries are expensive!

Type Conversions

// Narrowing conversions (potential data loss!)

}




// Widening conversions (always safe, just slow)

// To/from unsigned (bit pattern reinterpretation)

// To/from strings (for I/O and debugging)

let hex_str = to_hex_string(big_num);                    // "0x1E240..."

Modular Arithmetic

Critical for cryptography: 

// Modular exponentiation (square-and-multiply algorithm)


fn mod_exp(base: int2048, exponent: int2048, modulus: int2048) -> int2048 {
if modulus == 1 { return 0; }




let result: int2048 = 1;
let base_mod = base % modulus;
let exp = exponent;




while exp > 0 {
// If exp is odd, multiply result by base
if (exp & 1) == 1 {
result = (result * base_mod) % modulus;
}




// Square the base
base_mod = (base_mod * base_mod) % modulus;




// Divide exponent by 2
exp >>= 1;
}




return result;
}

Chinese Remainder Theorem (CRT) Optimization

// RSA decryption can be ~4× faster using CRT!


fn rsa_decrypt_crt(c: uint2048, d: int2048, p: uint2048, q: uint2048) -> uint2048 {
// Precompute (done once during key generation)
let dp = d % int2048(p - 1);  // d mod (p-1)
let dq = d % int2048(q - 1);  // d mod (q-1)
let q_inv = mod_inverse(int2048(q), int2048(p));  // q^(-1) mod p




// Decrypt modulo p and q separately (much faster!)
let m1 = mod_exp(c, dp, uint2048(p));  // c^dp mod p
let m2 = mod_exp(c, dq, uint2048(q));  // c^dq mod q




// Combine using CRT
let h = (q_inv * int2048(m1 - m2)) % int2048(p);
let m = m2 + uint2048(h) * q;




return m;
}




// Why CRT is faster:
// - Operates on ~1024-bit values instead of ~2048-bit
// - Modular exponentiation complexity is ~O(bits³)
// - Two 1024-bit exponentiations faster than one 2048-bit!
// - Speedup: ~4× for RSA-2048, ~4× for RSA-4096

Prime Number Generation

Essential for RSA key generation: 

// Generate a random large prime


fn generate_large_prime(bits: int) -> int2048 {
loop {
// Generate random odd number of specified bit length
let candidate = generate_random_odd(bits);




// Quick divisibility test (rule out ~99% of candidates quickly)
if has_small_prime_factor(candidate) {
continue;
}




// Miller-Rabin primality test (probabilistic)
if miller_rabin(candidate, rounds: 40) {
return candidate;
}
}
}




// Miller-Rabin primality test
fn miller_rabin(n: int2048, rounds: int) -> bool {
if n < 2 { return false; }
if n == 2 || n == 3 { return true; }
if n % 2 == 0 { return false; }




// Write n-1 as 2^r * d
let d = n - 1;
let r = 0;
while d % 2 == 0 {
d /= 2;
r += 1;
}




// Run multiple rounds with random bases
till(rounds - 1, 1) {
let a = random_range(2, n - 2);
let mut x = mod_exp(a, d, n);




if x == 1 || x == n - 1 {
continue;
}




let mut composite = true;
till(r - 2, 1) {
x = mod_exp(x, 2, n);
if x == n - 1 {
composite = false;
break;
}
}




if composite {
return false;  // Definitely composite
}
}




return true;  // Probably prime (error rate: 4^(-rounds))
}

4. Constant-Time Cryptographic Operations

// CRITICAL: Timing attacks can leak private keys!


// ✗ BAD: Branching on secret data (leaks information via timing!)
fn mod_exp_insecure(base: int2048, exp: int2048, mod: int2048) -> int2048 {
let result = 1;
let bits = get_bits(exp);
till(bits.length - 1, 1) {
if bits[$] == 1 {  // ← TIMING LEAK! Branch depends on secret exp!
result = (result * base) % mod;
}
base = (base * base) % mod;
}
return result;
}




// ✓ GOOD: Constant-time implementation (no secret-dependent branches)
fn mod_exp_secure(base: int2048, exp: int2048, mod: int2048) -> int2048 {
let result = 1;
let base_pow = base % mod;
let bits = get_bits(exp);




till(bits.length - 1, 1) {
// Always do both operations, select result without branching
let result_if_0 = result;
let result_if_1 = (result * base_pow) % mod;




result = constant_time_select(bits[$], result_if_1, result_if_0);
base_pow = (base_pow * base_pow) % mod;
}




return result;
}
