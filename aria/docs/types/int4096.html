int4096 - 4096-bit Integer (RSA-16384 Maximum)

Category: Types → Large Integers / Cryptography Purpose: Maximum-security cryptography, RSA-16384 operations Status: ✅ COMPLETE (All operations implemented and tested)

Overview

int4096 is a 4096-bit signed integer type representing the MAXIMUM size integer in Aria. It's designed for extreme-paranoia cryptographic scenarios requiring RSA-16384 (the practical upper limit for RSA key sizes). ⚠️ IMPORTANT: This is the largest integer type. There is no int8192. If you need larger, you must use arbitrary-precision libraries (non-deterministic).

Key Characteristics

| Property | Value | |----------|-------| | Total Bits | 4096 (64 × 64-bit limbs) | | Range (signed) | -2^4095 to +2^4095 - 1 | | Range (unsigned) | 0 to 2^4096 - 1 | | Storage | 512 bytes (64 limbs) | | ERR Sentinel | limbs[63]=0x8000000000000000, others=0 | | Endianness | Little-endian limbs |

uint64:limbs[64];  // Little-endian: limbs[0] = LSB, limbs[63] = MSB
};

// Limb 0: bits 0-63
// Limb 1: bits 64-127
// ...
// Limb 63: bits 4032-4095 (sign bit in bit 4095)

ERR Sentinel Pattern

ERR Value

// ERR = minimum signed value (1 << 4095)


// Limbs: [0, 0, 0, ..., 0, 0x8000000000000000]
//        ^^^^^^^^^^^^^^^^^^^^-64 limbs-^^^^

// Same pattern as int1024, int2048
// Cannot be produced by valid arithmetic

ERR Propagation

int4096:still_err = bad + good;    // ERR (sticky!)
int4096:also_err = good * bad;     // ERR
int4096:div_err = good / bad;      // ERR




// ERR sticky until explicitly checked and handled

Cryptographic Operations

RSA-16384 Scenario (Extreme Security)

When would you use RSA-16384?
  • National security archives (100+ year confidentiality)
  • Nuclear command & control systems
  • Long-term medical genomic data
  • Diplomatic secrets & treaty documents
// RSA-16384: Two 8192-bit primes


// (int4096 can hold 4096-bit values, so partial operations)




// Generate 4096-bit prime candidates
int4096:prime_candidate = generate_random_4096();




// Miller-Rabin primality test (simplified)
int4096:witness = 2i4096;
int4096:exponent = prime_candidate - 1i4096;




int4096:test_result = mod_exp_4096(witness, exponent, prime_candidate);




if (test_result == ERR) {
stderr.write("Primality test computation failed\n");
!!! ERR_CRYPTO_PRIMALITY_TEST_FAILED;
}




// Fermat's Little Theorem: a^(p-1) ≡ 1 (mod p) if p is prime
if (test_result != 1i4096) {
// Definitely composite, try next candidate
prime_candidate = prime_candidate + 2i4096;
}

Modular Exponentiation (4096-bit)

// Core RSA operation: base^exponent mod modulus


// For 4096-bit values (64-limb computation!)




function mod_exp_4096(
int4096:base,
int4096:exponent,
int4096:modulus
) : int4096 begin
int4096:result = 1i4096;
int4096:current_base = base % modulus;




till exponent == 0i4096 loop
// If exponent is odd
int4096:is_odd = exponent & 1i4096;
if (is_odd == 1i4096) {
result = (result * current_base) % modulus;




if (result == ERR) {
stderr.write("Modular multiplication overflow\n");
return ERR;
}
}




// Square and halve
current_base = (current_base * current_base) % modulus;
exponent = exponent >> 1;




if (current_base == ERR || exponent == ERR) {
return ERR;
}
end




return result;
end




// RSA decryption with 4096-bit values
int4096:ciphertext = ...;
int4096:private_exponent = ...;
int4096:modulus = ...;




int4096:plaintext = mod_exp_4096(ciphertext, private_exponent, modulus);




if (plaintext == ERR) {
// Decryption failure - corrupted key or ciphertext
!!! ERR_CRYPTO_DECRYPTION_FAILED;
}

Nikola Use Cases

Why int4096 for Nikola?

Century-Scale Security: Some therapeutic data must remain confidential for a patient's entire lifetime plus legal retention periods (potentially 100+ years). RSA-16384 provides security margin against future attacks.

Ultra-Long-Term Medical Archive

// Nikola stores neurological baselines for rare genetic conditions


// This data must remain secure for decades (potential research value)




int4096:archive_prime_p = load_archive_prime("century_p.key");
int4096:archive_prime_q = load_archive_prime("century_q.key");




if (archive_prime_p == ERR || archive_prime_q == ERR) {
stderr.write("CRITICAL: Century-scale encryption keys corrupted\n");
!!! ERR_CRYPTO_CENTURY_KEY_FAILED;
}




// Modulus for RSA-16384 (two 8192-bit primes)
// (int4096 can represent one prime, full RSA needs higher level)
int4096:archive_modulus_low = archive_prime_p * archive_prime_q;  // Low bits




// Encrypt baseline neurological snapshot
int4096:baseline_hash = hash_neurological_state();
int4096:public_exp = 65537i4096;




int4096:encrypted_baseline = mod_exp_4096(
baseline_hash,
public_exp,
archive_modulus_low
);




if (encrypted_baseline == ERR) {
// CRITICAL FAILURE: Cannot secure patient data
// Better to fail explicitly than risk data breach
stderr.write("CRITICAL: Baseline encryption failed\n");
!!! ERR_CRYPTO_BASELINE_ENCRYPTION_FAILED;
}

// ⚠️ WARNING: int4096 is EXPENSIVE memory-wise


// This is 5 KB of stack:

int4096:b;

int4096:e;
int4096:f;

Division by Zero (Fail-Hard)

int4096:numerator = 1000i4096;


int4096:denominator = 0i4096;




int4096:result = numerator / denominator;  // ERR sentinel




// ALWAYS check critical operations
if (result == ERR) {
!!! ERR_DIVISION_BY_ZERO;
}

Overflow in Multiplication

// CRITICAL: int4096 multiplication produces 8192-bit result


// But int8192 does NOT exist! Result is truncated.




int4096:large_a = MAX_INT4096 / 2;
int4096:large_b = 3i4096;

Performance Characteristics

| Operation | Complexity | Rough Cycles (x86-64) | |-----------|------------|----------------------| | Addition | O(64) limbs | ~80-120 cycles | | Subtraction | O(64) limbs | ~80-120 cycles | | Multiplication | O(4096) limb-ops | ~5000-10000 cycles | | Division | O(4096) limb-ops | ~10000-20000 cycles | | Comparison | O(64) limbs | ~80-120 cycles | | Shifts | O(64) limbs | ~80-150 cycles |

Real-World Context (3 GHz CPU):
  • int4096 addition: ~40 nanoseconds
  • int4096 multiplication: ~3 microseconds
  • int4096 division: ~6 microseconds
For Nikola's <1ms timestep:
  • Can do ~170 int4096 divisions per frame
  • Can do ~330 int4096 multiplications per frame
  • Can do ~25,000 int4096 additions per frame
⚠️ MEMORY BANDWIDTH: 512 bytes per value means memory bandwidth becomes bottleneck faster than computation.

  • Portability: Compiles on embedded systems (future: ARM, RISC-V)

stderr.write("✅ RSA-16384 key parameters validated\n");

When NOT to Use int4096

❌ Don't use int4096 if:
  • int2048 or int1024 would suffice (99% of cases)
  • You need arbitrary precision (use external library)
  • Memory is constrained (<10 KB stack)
  • You're doing array operations (512 bytes × array size!)
✅ DO use int4096 for:
  • RSA-16384 operations (national security)
  • Century-scale data encryption
  • Genomic privacy (rare genetic conditions)
  • Maximum-paranoia cryptographic scenarios
Rule of thumb: Start with int1024. Move to int2048 if needed. Only use int4096 when you have a specific, documented threat model requiring RSA-16384.
